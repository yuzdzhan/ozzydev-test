Build a RAG Chatbot in an Afternoon

O OzzyDev Team Jun 12, 2026 8 min read

Large language models are brilliant generalists and terrible specialists. Ask one about your product docs, your internal wiki, or last week's release notes, and you'll get a confident answer that may be entirely invented. Retrieval-augmented generation (RAG) fixes this by giving the model the relevant facts at question time — so it answers from your data instead of its imagination.

Here's the entire pattern in one sentence: find the most relevant chunks of your documents, paste them into the prompt, and ask the model to answer using only those chunks. Let's build it.

The four moving parts

  1. Ingest — split documents into bite-sized chunks.
  2. Embed — turn each chunk into a vector and store it.
  3. Retrieve — embed the user's question, find the nearest chunks.
  4. Generate — hand those chunks to the LLM and ask it to answer.

1. Chunk your documents

Models have limited context and retrieval works best on focused passages. A good default is ~500–800 tokens per chunk with a little overlap so you don't slice a sentence in half. 

function chunk(text, size = 600, overlap = 80) {
  const words = text.split(/\s+/);
  const chunks = [];
  for (let i = 0; i < words.length; i += size - overlap) {
    chunks.push(words.slice(i, i + size).join(" "));
  }
  return chunks;
}

2. Embed and store

An embedding turns text into a list of numbers that captures meaning. Similar text lands near similar text. Store the vectors in any vector database — or, for a small project, an in-memory array is genuinely fine. 

// Pseudocode — works with any embeddings provider
const store = [];
for (const c of chunks) {
  const vector = await embed(c);        // returns number[]
  store.push({ text: c, vector });
}
Tip: Don't reach for a managed vector DB on day one. Start in memory, prove the idea works, then scale to a real store (pgvector, Qdrant, Pinecone…) when you actually have the volume to justify it.

3. Retrieve the closest chunks

Cosine similarity is the standard distance metric. Embed the question, score every chunk, and keep the top few. 

function cosine(a, b) {
  let dot = 0, na = 0, nb = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    na += a[i] * a[i];
    nb += b[i] * b[i];
  }
  return dot / (Math.sqrt(na) * Math.sqrt(nb));
}

async function retrieve(question, k = 4) {
  const q = await embed(question);
  return store
    .map(item => ({ ...item, score: cosine(q, item.vector) }))
    .sort((a, b) => b.score - a.score)
    .slice(0, k);
}

4. Generate a grounded answer

The magic is in the prompt. Tell the model to answer only from the provided context and to admit when it doesn't know. 

async function answer(question) {
  const hits = await retrieve(question);
  const context = hits.map((h, i) => `[${i + 1}] ${h.text}`).join("\n\n");

  const prompt = `Answer the question using ONLY the context below.
If the answer isn't there, say "I don't know."
Cite sources like [1], [2].

Context:
${context}

Question: ${question}`;

  return await llm(prompt);   // your model call
}

That's it. Roughly fifty lines and you have a chatbot that answers from your own documents with citations. No fine-tuning, no GPU, no PhD.

Where to go next

  • Re-ranking — run a cross-encoder over the top 20 hits to sharpen the top 4.
  • Hybrid search — combine keyword (BM25) with vector search for names and IDs.
  • Evaluation — build a tiny set of question/answer pairs and measure before you tweak.
RAG isn't a product — it's plumbing. Get the retrieval right and the model almost takes care of itself.
